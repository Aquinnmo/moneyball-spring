# Global Agent Instructions

## General Guidelines
- **Context Efficiency:** Try to rely on `AI.md` and `ARCHITECTURE.md` to navigate the workspace. Avoid overly broad codebase searches when you know the specific domain files.
- **Formatting:** Keep explanations minimal and focus on exact tool usage. Provide concise diffs and write complete executable sections when generating code.

## Code Conventions (Kotlin & Spring)
- **Serialization:** Always use `@JsonIgnoreProperties(ignoreUnknown = true)` on new DTOs/data classes that map to external APIs (StatsAPI, Statcast). This prevents breakage when external schemas inevitably evolve.
- **Language:** Use idiomatic Kotlin (e.g., `?.let`, `map`, `flatMap`, `data class`). Avoid Java-isms.
- **Nullability:** Strict adherence to Kotlin's null safety. Use nullable types `?` for external API fields since MLB APIs frequently omit fields (e.g. `String?`, `Double?`).
- **Logging:** Prefer utilizing SLF4J or standard Kotlin `println` wrappers as strictly established in the current codebase conventions.

## Editing & Refactoring
- **Acceptable Refactors:** Group related data structures in the `structures/` package. Ensure DTOs do not contain business logic.
- **Documentation:** Use concise KDoc (`/** */`) for complex data transformations, specifically when calculating custom MLB heuristics (e.g., `delta_pitcher_run_exp`).

## API Contracts
- Before modifying existing endpoint signatures, check `API_CONTRACT.md`. Any change to REST controllers MUST be synchronized with `API_CONTRACT.md`.
